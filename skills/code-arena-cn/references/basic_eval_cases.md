# 基线评测用例(静态,所有项目通用)

与 prompt 无关、适用于**任何**受评 Web 应用实现的评测用例。本 rubric 为静态:不引用任何特定需求、特性集或项目。独立产出 **/100** 分,衡量通用工程质量(运行时安全、视觉 UX 质量、代码质量、开源治理),另加最高 **+10** 加分。

每个实现得到本 Basic 分(/100)**和**一份来自该项目自身用例文件的需求专属分(/100);报告同时呈现两者。

| 域 | 分值 |
|---|---:|
| B1. 运行时安全 | 30 |
| B2. 前端 UX — 视觉质量 | 18 |
| B3. 代码质量(SAST + 架构 + 可读性) | 30 |
| B4. 开源治理 | 10 |
| B5. 测试覆盖率 | 12 |
| **合计** | **100** |
| 加分(动态) | 最高 +10 |

*(Rubric 说明,2026-06-26:新增 B5 测试覆盖率(12)并重新配比——开源治理右调 16 → 10(仅两项近乎二元的检查),代码质量 36 → 30;B1/B2 不变。本日期之前评测的轮次用 B3 = 36、B4 = 16、无 B5。)*

## 1. 范围:按目标(仓库 + 分支)解析

一个目标是一对 *(仓库, 分支)*。先解析出在范围内文件集,再在代码审查、SAST、指标、架构、治理、覆盖率评分中**只评该文件集**:

- **默认当前分支**(未指定分支):工作树**有未提交改动** → 范围 = 相对 `HEAD` 的**未提交增量**(未跟踪 + 已修改,历史默认);工作树**干净** → 范围 = **最后一次提交的内容**(尖端改动的文件;若尖端为根/`init` 提交则取完整树)。
- **显式指定的分支**(一个或多个):范围 = 该分支**尖端(最后一次提交)**的内容;工作树未提交改动忽略。同一仓库多分支各在**独立临时检出**中评测(`git worktree add` / `git archive`)。

**先解析项目。** 若目标以名称给出,在工作根目录下定位(名称唯一):`REPO=$(find <工作根目录> -type d -name '<name>' -not -path '*/node_modules/*' | head -1)`。下文 `<repo>` 用解析出的 `$REPO` 路径。

**范围闸(每个目标先跑):**

```
git -C <repo> rev-parse --abbrev-ref HEAD        # 当前分支(默认目标)
git -C <repo> status --porcelain                 # 空 = 干净
# 默认当前分支且有改动 → 未提交增量:
git -C <repo> ls-files --others --exclude-standard   # 未跟踪
git -C <repo> diff --name-only HEAD                  # 相对 HEAD 已修改
# 干净的当前分支,或显式指定的 <ref> → 最后一次提交内容:
git -C <repo> diff --name-only <ref>~1 <ref>         # 尖端改动的文件
git -C <repo> rev-list --count <ref>                 # 为 1 ⇒ 根提交 ⇒ 取 <ref> 完整树
```

将得到的在范围内文件集记入报告。以下所有静态分析仅限于此。

## 2. 方法与标准工具(固定,确保每个项目判定一致)

| 方法 | 标准工具 | 用法 |
|---|---|---|
| 运行时安全探测 | 评估方自研 HTTP 探测 harness | 后端在专用端口启动;请求/响应记为 JSON。独立于项目自带测试。 |
| UI 视觉采集 | Playwright(headless Chromium)在桌面 1280×900、平板 768×1024、移动 390×844 的全页截图 + 对照 rubric 的多模态评审 | 逐张查看并评分;缺陷以截图文件为证。 |
| SAST(静态安全) | **Semgrep**(官方 `semgrep-rules`,`javascript/` + `security/` 集)+ `npm audit` | 扫在范围内源码;发现按严重度分桶并附规则 ID + `file:line`。 |
| 架构质量 | **dependency-cruiser** | 从 JSON 输出统计应用级循环依赖与"逆向"(低层→高层)import;grep store 层之外的直接持久化访问。 |
| 可读性 / 复杂度 | **ESLint**(+ `eslint-plugin-complexity`)+ **jscpd**(重复率) | 圈复杂度、函数长度、LOC、重复率% —— 并排呈现。 |
| RESTful API 设计与兼容性 | **评估方 API 探测 + 人工路由/控制器审查** + **`git diff`** | 动词/状态码/资源路径约定;兼容性用未提交 vs `HEAD`(或 `HEAD` vs `HEAD~1`)diff——无快照文件。 |
| 开源治理 | **license-checker** + `npm audit --omit=dev` | 依赖许可证兼容性(仅 OSI 宽松型)+ 依赖无 High/Critical CVE。 |
| 测试覆盖率 | **`c8`**(V8 覆盖率,可包裹任意 runner)+ 前端 **`vitest --coverage`** | 在覆盖率下跑项目自带套件(`c8 <test-cmd>`);报告前后端在范围内代码的分支/行/函数覆盖率%;把测试映射到端点/角色/错误路径/核心场景;核查已提交的 E2E(Playwright/`playwright-cli`)写操作流程覆盖。输出 `lcov` + 汇总到 `evidence/coverage/`。 |
| 代码审查 | 人工查阅(grep / 读文件) | 以 `file:line` 为证。仅限未提交文件集。 |

**独立性说明:** 所有运行时/行为/视觉**正确性**分数来自评估方自研、对运行中应用的探测——绝不用项目自带测试套件评判应用自身正确性。**B5 测试覆盖率是唯一审视项目自带套件之处**,且仅用于评判*测试本身的彻底程度*(覆盖了多少代码/行为),绝不用于评判应用是否正确。这与独立性规则相容。

**评分图例:** Pass = 满分;Partial = 50%;Fail = 0。

# B3. 代码质量(30 分)

对在范围内代码跑标准质量工具链,再施加评审判断。子域:B3a SAST(10)、B3b 架构(10)、B3c 可读性(6)、B3d RESTful API 设计与兼容性(4)。*(测试覆盖率原先隐含,现独立为 B5 域。)*

## B3a. 静态安全分析 — SAST(10 分)

工具:**Semgrep**(官方 `semgrep-rules`,`javascript/` + `security/` 集)+ `npm audit`。

| ID | 项 | 分值 |
|---|---|---|
| B3a-1 | 应用代码无 High/Critical Semgrep 发现 | 6 |
| B3a-2 | 无应用代码可达的 High/Critical 依赖漏洞(`npm audit`) | 2 |
| B3a-3 | 少/无 Medium Semgrep 发现;若有则有正当理由或低影响 | 2 |

评分:从满分起;**每个独立 High/Critical** 应用发现 **−3**(子项下限 0),**每个 Medium −1**(有上限)。每个发现附规则 ID + `file:line`。与 B1 交叉核对避免同一问题重复扣分——SAST 计*静态可检出性*,B1 计*运行时可利用性*。

## B3b. 架构与结构 — 高内聚 / 低耦合 / 高可扩展(10 分)

工具:**dependency-cruiser**(分层/循环/耦合图)+ 对记录信号的人工审查。对新(未提交)代码以三大经典结构质量判定。

| ID | 项 | "好"的样子 | 分值 |
|---|---|---|---|
| B3b-1 | 高内聚 + 关注点分离 | 清晰分层(如 routes → controllers → services/repositories;客户端 views/stores/api 分离);每模块单一聚焦职责;路由处理器内无业务逻辑;无巨石文件 | 3 |
| B3b-2 | 低耦合 | 模块依赖抽象而非具体内部;持久化隔离在 repository/store 层之后(换存储后端只动少量文件);配置/密钥注入式;无循环依赖;客户端用单一 API client 而非散落 fetch | 4 |
| B3b-3 | 高可扩展 / 可演进 | 新增实体、角色、路由或规则遵循显见既有范式;分页/过滤为更大数据集设计;无阻碍扩展的硬编码假设;无状态请求处理 | 2 |
| B3b-4 | 一致性与约定 | 命名、错误处理范式、响应结构一致 | 1 |

**可复现信号(每个项目记录):** 层间依赖方向(有无"逆向"import?)、持久化是否仅经 store 层可达(grep 其外的直接文件系统/DB 访问)、每模块 import 扇入/扇出、是否存在循环 import,以及一处代表性变更(加字段/加角色)会触及多少文件。

## B3c. 可读性、简洁性与可维护性(6 分)

工具:**ESLint**(+ `eslint-plugin-complexity`)+ **jscpd**。指标:文件 LOC、平均/最大函数长度、圈复杂度、重复率%。

| ID | 项 | "好"的样子 | 分值 |
|---|---|---|---|
| B3c-1 | 简洁 vs 冗长 | 无多余样板或复制粘贴;DRY;简洁但不晦涩;无死代码/过度设计 | 2 |
| B3c-2 | 可读性 / 可理解 | 命名清晰、函数短小、嵌套浅;自上而下可读;注释解释"为什么"而非噪声 | 3 |
| B3c-3 | Lint、复杂度与重复率预算 | ESLint 通过(或仅琐碎警告);无函数超出合理预算(圈复杂度 ≤ ~10、函数 ≤ ~60 行);**代码重复率 ≤ 3%**(jscpd) | 1 |

**评分指引:** B3b 与 B3c 按项 Pass/Partial/Fail,以并排记录的指标(LOC、最大函数长度、最大复杂度、ESLint 警告、重复率%、Semgrep 计数)为锚。**重复率阈值为 ≤ 3%**(jscpd):≤3% 通过 B3c-3 的重复率部分;>3% 使 B3c-3 至多 Partial,显著偏高则 Fail。简洁性相对*任务*判断:更大但凭真正稳健性配得上其体量的代码库不被扣分;小而晦涩/重复的也不白给分。

## B3d. RESTful API 设计与兼容性(4 分)

工具:**评估方 API 探测 + 对路由/控制器的人工审查**,兼容性检查另用 **`git diff`**。以观测到的 HTTP 面(动词、状态码、资源路径、请求/响应结构)判定。

| ID | 项 | "好"的样子 | 分值 |
|---|---|---|---|
| B3d-1 | RESTful 设计规范 | 面向资源、复数名词路径(`/api/reimbursements/:id`);动词语义正确(GET 读 / POST 建 / PUT-PATCH 改 / DELETE 删);状态码有意义(200/201/204/400/401/403/404/409,非一律 200);无状态;一致的 JSON 请求/响应结构;集合端点支持分页/过滤查询参数 | 2 |
| B3d-2 | API 前向/后向兼容性 | API 为不破坏客户端而设计:版本化或稳定契约(如 `/api/v1` 或显式版本策略)、字段增量/可选、容忍性读取(忽略未知字段)、字段名与响应结构稳定、无对既有字段或状态码的破坏性删改 | 2 |

### 兼容性检查方法(无快照)

B3d-2 按两轴评估:

1. **设计层面能力(始终):** 审查代码是否具备上述兼容友好特性(版本化方案、字段增量/可选、容忍性读取、稳定结构)。
2. **历史 diff(git 历史允许时,无快照文件):** 用 `git diff` 在 routes/controllers/DTO 上检测破坏性 API 变更——
   - 若工作区相对一个有意义的前次提交有未提交改动:**未提交工作区 vs 最近提交(`HEAD`)**;
   - 否则:**最近提交 vs 上一次提交(`HEAD` vs `HEAD~1`)**。
   破坏性变更 = 删除/重命名端点、删除/重命名请求或响应字段、动词/状态码语义改变、或收紧必填/校验导致旧有合法请求被拒。
   若无可对比的前一版本(如 `HEAD` 为空/`init`、整个实现都未提交),则只按设计层面评 B3d-2,并记"无历史接口可 diff"。

> **轮次隔离的有限例外:** B3d-2 可对比当前接口与**同一仓库**的前一 **git** 版本以检测破坏性变更。这是项内技术 diff,不是跨轮分数/叙述对比——以独立的兼容性发现陈述(不写"相比第 X 轮回退")。

# B4. 开源治理(10 分)

检查项目引入的开源依赖。仅两项检查。工具:**license-checker**(许可证清单)+ **`npm audit --omit=dev`**(依赖 CVE)。

| ID | 项 | "好"的样子 | 分值 |
|---|---|---|---|
| B4-1 | 依赖许可证兼容性 | **全部**生产依赖均为 OSI 认可的**宽松型**许可证(MIT / BSD / Apache-2.0 / ISC 等)。无强 copyleft(GPL / AGPL / LGPL / SSPL / MPL),无 `UNKNOWN`/`UNLICENSED` 依赖。 | 5 |
| B4-2 | 无 High/Critical 依赖 CVE | `npm audit --omit=dev` 在生产依赖中报告**零 High、零 Critical** 公告。 | 5 |

**评分:**
- B4-1:每个生产依赖都 OSI 宽松则满 5。**每个独立 copyleft 或未知许可依赖 −2.5**(下限 0)。项目自身 `package.json` 的 `license` 字段**不**评估——只看所引入开源组件的许可证。
- B4-2:无 High/Critical CVE 则满 5。**每个独立 High/Critical 公告 −2.5**(下限 0)。Moderate/Low 公告记录但不计分。

> B4-2 与 B3a-2 都用 `npm audit`。为避免重复计算:B3a-2(SAST)计*应用代码可达*的依赖风险;B4-2 计*整体依赖 CVE 态势*。报告交叉引用两者,且单一域内不对同一公告重复扣分。

# B5. 测试覆盖率(12 分)

项目**自带**的已提交测试套件对其代码与所需行为的覆盖彻底程度。这评判的是*测试的彻底程度*,不是应用的正确性(功能正确性只由评估方探测判定——见独立性说明)。仅在**在范围内代码**上测量。

工具:**`c8`**(V8 覆盖率;可包裹任意 runner——`c8 <test-cmd>`,如 `c8 npm test` 跑 `jest` / `node:test` / `vitest` / `mocha`),前端用 **`vitest --coverage`**,外加对测试触及内容的需求映射审查。`lcov` + 汇总存 `evidence/coverage/`。

12 分在原始覆盖率%(B5-1/-2 = 5.5)与需求/E2E 覆盖(B5-3/-4/-5 = 6.5)之间**均衡**。

| ID | 项 | "好"的样子 | 分值 |
|---|---|---|---|
| B5-1 | 后端代码覆盖率 | `c8` 跑后端套件:**分支 ≥ 70% 且 行 ≥ 80%**(在范围内代码) | 3.5 |
| B5-2 | 前端逻辑覆盖率 | `vitest --coverage`/`c8` 跑前端**逻辑**(stores / api / utils,非 `.vue` 模板):**行 ≥ 60%**,或由已提交 E2E 等价覆盖 | 2 |
| B5-3 | 错误/负路径覆盖 | 测试断言失败路径,而非只有正常路径:覆盖代表性的 **400 / 401 / 403 / 404 / 409** 用例 | 2 |
| B5-4 | 核心场景与角色覆盖 | 核心流程(登录、账号 CRUD、提交、审批/驳回、撤销)有正常路径测试,且覆盖**管理员与员工两种角色** | 1.5 |
| B5-5 | 已提交 E2E UI 写操作流程覆盖 | 已提交的 E2E 测试(Playwright / `playwright-cli` 或等价)经 UI 端到端驱动**写操作流程**:创建员工、提交、审批、驳回、撤销 | 3 |

### 评分(B5)

- **中等阈值。** B5-1:分支 ≥ 70% 且 行 ≥ 80% 满分;分支 ≥ 50% **或** 行 ≥ 65% 为 **Partial(50%)**;以下为 **Fail(0)**。B5-2:逻辑模块行 ≥ 60%(或等价已提交 E2E)满分;≥ 40% 为 Partial;否则 Fail。
- **B5-3 / B5-4 / B5-5** 按已提交测试实际覆盖所列用例/流程的比例判 Pass / Partial(50%)/ Fail(如 5 个写操作流程的 E2E:全覆盖=满分,部分=Partial,一个都没提交=Fail)。
- **套件无法运行 → B5 = 0**(覆盖率确实不可测)。记录原因(如缺 `package.json`/测试脚本)。套件须能从文档命令运行;报错的套件按其可测覆盖率计(往往为 0)。
- **报告**前后端的分支/行/函数覆盖率%并排于静态分析汇总,附各 B5 项判定。

### 不重复计算(B5)

- **B5 vs 轮次测试域(R4)。** R4(在 round 文件)评判测试**存在、可运行、通过**且有文档;**B5 评判其覆盖多少**(深度)与已提交 E2E 触及哪些**写操作流程**(广度)。存在/通过 ≠ 覆盖,属不同维度。Basic 与 Round 本就是两套独立 /100,弱套件在两边都失分是设计使然;但不要在*同一*份 rubric *内*对同一缺口扣两次。
- **B5-5 vs R4-3。** R4-3 计前端/E2E 测试脚本*存在*;B5-5 计已提交 E2E 实际覆盖**多少所需写操作流程**。交叉引用;不要在 Basic 内对同一缺失脚本重复扣分。
- **B5 vs B3c。** B3c 评源码的可读性/复杂度/重复率;B5 评测试套件的触及面。两者独立。

# 加分 — 超出基线工程的额外项(最高 +10,动态)

**加分列表为动态。** 下为小型静态种子。评测中,审查每个实现后,评估方须:

1. **新增**上文未计的、新发现的有价值工程额外项(附拟议分值)。
2. **删除**不相关或框架轻易自带的种子项。
3. 记录最终应用列表(每项附证据),为公平**对所有受评项目一致核对**——在一个项目发现的加分项也要在其他项目核查。

每项 +1 到 +2,总计上限 +10。

### 静态种子列表

只计真正超出所有计分项的额外项。**任何已在某域计分的内容不得作加分**(如魔数上传校验 = B1-5;fail-fast 密钥 = B1-8;状态标签 = B2-5;一致错误处理 = B3b-4)——否则重复计算。

| 额外项 | 典型分值 |
|---|---|
| 认证暴力破解 / 限流 | +2 |
| 安全头(如 helmet) | +1 |
| 敏感操作审计日志 | +2 |
| 刷新令牌 / 会话轮换 | +2 |
| 超出必需屏幕的统计仪表盘 / 概览视图 | +1 |
| 随代码提交的 CI / 自动化质量门禁 | +1 |

## 执行标准(共享)

- 先过范围闸(§1);记录每个项目的在范围内文件集。
- 每个后端在临时副本的专用端口运行;不修改原始仓库。
- 运行时/安全/UX 探测使用评估方自研脚本。
- 每个 API 探测把请求 + 响应状态/体记为 JSON 证据。
- 视觉 UX 在桌面/平板/移动对应用关键页面截全页图;逐张评审并引用文件。
- SAST:`semgrep --config <path-to>/semgrep-rules/javascript --config <path-to>/semgrep-rules/<lang>/security` 扫在范围内源码(官方 `semgrep-rules`,克隆),外加 `npm audit --omit=dev`。
- 代码指标:ESLint + 指标采集(文件 LOC、最大/平均函数长度、最大圈复杂度、重复率)+ 耦合信号(跨层 import、store 层外直接持久化访问、循环依赖、代表性变更触及文件数)—— 并排。
- **有效代码行:** 取 cloc `code` 行(排除空行+注释),覆盖**全部在范围内源码**——JS/TS、Vue(`<template>`+`<script>`)、CSS/SCSS、HTML。**产品代码与测试代码分开报告**(两个数字)。两遍:
  - 产品代码:`cloc <在范围内> --exclude-dir=node_modules,dist,build,coverage --not-match-f='(package(-lock)?\.json|.*\.config\.[jt]s|tsconfig.*json|\.?eslintrc.*|.*\.lock|db\.json|.*(seed|fixture|mock).*|.*\.(test|spec)\.[jt]sx?)$' --json` → 取 `code` 合计。
  - 测试代码:`cloc <测试文件:*.test.*、*.spec.*,及 test/ tests/ __tests__/ 目录> --json` → 单独报告其 `code` 合计。
  绝不计空行、注释、配置文件、数据/种子/lock 文件、生成物或 `node_modules`。
- RESTful API 设计与兼容性:从路由 + 探测证据列出 HTTP 面(动词、路径、状态码);兼容性跑 `git diff`(未提交 vs `HEAD`,否则 `HEAD` vs `HEAD~1`)于 routes/controllers/DTO 上找破坏性变更——无快照文件。若无前一版本,仅评设计层面兼容性。
- 治理:`license-checker --production --summary`(标记任何非 OSI 宽松 / UNKNOWN / UNLICENSED 依赖)+ `npm audit --omit=dev`(读 `.metadata.vulnerabilities` 看 High/Critical)。
- **测试覆盖率(B5):** 在覆盖率下跑项目自带套件——后端 `c8 --reporter=lcov --reporter=text <test-cmd>`(如 `c8 npm test`),前端 `vitest run --coverage`(或 `c8`)——扫在范围内代码;分支/行/函数覆盖率%记入 `evidence/coverage/`。再把已提交测试映射到端点/角色/错误路径(4xx)/核心场景,并核查已提交 E2E(`playwright-cli`/Playwright)覆盖了哪些 UI 写操作流程。若套件无法从文档命令运行,B5 = 0(记录原因)。覆盖率评判*测试的彻底程度*,绝非应用正确性。
- 此域集产出一个 **/100 Basic 分** + 加分,与需求专属分并列报告。
