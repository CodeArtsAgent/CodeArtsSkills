# Judge Codebase Evaluation Runs

Run in a new Judge agent context after every evaluated product has completed execution. Do not run inside an evaluated product's execution session or expose the sealed package to an evaluated product.

## J0: mandatory structured intake

Make the first tool call after loading this workflow a structured user-input call. Collect and explicitly confirm:

- round ID as the only shared-layout directory-selection input for public cases, product results, and report output; fix the evaluation root to `.codebase-eval`;
- matching sealed Judge package as a Judge-workspace-relative path when it was not supplied through a protected external input;
- the decision the report must support, report audience, and policy for missing, unfinished, protocol-error, infrastructure-error, or materially incomplete evidence;
- only when artifacts cannot decide them: product-label overrides, token comparability, and unresolved Judge-environment actions.

Discover approved public packages only under `.codebase-eval/<round-id>/evalset`, discover raw runs under direct non-reserved product children, generate a Judge run ID, and derive the report root as `.codebase-eval/<round-id>/report/<judge-run-id>`. Include every validated system and case present in the round. Require all inputs to share the same `roundId`; never compare across rounds silently. Do not ask for an evaluation root, public-package paths, raw-run paths, an independent report path, a Judge run ID, or a system subset. Never search for a latest evalset or inspect evalset-like artifacts outside `.codebase-eval`. Do not ask for repository profiling, source authorization, case design, execution concurrency, timeout, snapshot policy, adapters, or retry limits. Derive product identities and run facts from validated raw results. Do not infer missing products or silently drop cases. Ask before any dependency or environment mutation.

Derive any application-under-evaluation credentials needed by private result checks from the sealed package and its pre-authoring verification records. Do not ask a general credential-availability question and do not request credentials for the evaluated coding product. Re-run least-privilege, redacted readiness checks before judging; stop and request remediation only for a concrete failed requirement. Never persist secret values, and clean all Judge-created sessions or temporary credential state.

After the user confirms the round ID, run the resolver from the evaluated repository root with a relative path to the bundled script and `--round <round-id>`. Use its discovered relative public-package, raw-run, and report paths verbatim. If multiple comparison cohorts exist, ask which cohort to judge; do not ask the user to type its paths.

After confirmation, write `judge-intake.json` from [../../assets/judge-intake.template.json](../../assets/judge-intake.template.json) under the Judge output root.

Read [../../references/output-layout.md](../../references/output-layout.md), [../../references/scoring-criteria.md](../../references/scoring-criteria.md), [../../references/case-schema.md](../../references/case-schema.md), [../../references/bilingual-output.md](../../references/bilingual-output.md), [../../references/environment-policy.md](../../references/environment-policy.md), and [../../references/data-lifecycle.md](../../references/data-lifecycle.md) completely before judging.

## Judge protocol

1. Verify the shared round layout, require each raw run at `.codebase-eval/<round-id>/<system-id>/<execution-run-id>`, and reject a run from another round, under `evalset/` or `report/`, or under the wrong product directory. Then verify H4 approval, private/public manifest identity including `roundId`, the exact same execution-contract SHA-256 across products, case-ID equality, base revisions, prompt locale comparability, environment baseline evidence, raw-run validation, and per-case evidence-manifest completeness. Report evidence gaps without silently treating absent evidence as success.
2. Require Execute v5 results with embedded validation. Verify that every compared product has one terminal execution result per attempted case, compact `projectRestored` and `externalStateClean` integrity status, `rollbackIntegrity: true`, zero rollback errors, and an existing retained completed-state snapshot for every non-protocol result. Capture, restoration, and cleanup result files are intentionally transient and must not be required by Report. Report never performs, retries, or evaluates rollback mechanics. Surface false integrity status as an execution-integrity failure under the confirmed incomplete-run policy; never silently mix Execute schema versions or treat a contaminated run as comparable.
3. Work only from retained completed-state snapshots. Run automated or human-reviewed result checks from the sealed package against Judge-only copies after candidate execution is complete. Do not alter the retained evidence. Use `run-verification.mjs` when the private case has an executable result check.
4. Apply the private scoring criteria and score limits, then apply the deterministic duration adjustment from the immutable Execute `durationMs` and the execution-contract `perCaseTimeoutMs`: no deduction at or below 25% of the limit, deduct 0.5 above 25% through 50%, and deduct 1 above 50%, at timeout, or when unfinished. Produce exactly one resulting `score` from 0 to 10 in 0.5 increments for each `system × case`. Do not create codebase, implementation, or speed sub-scores. Never replace the recorded duration with a Judge estimate.
5. Record criterion evidence and result-check findings bilingually. Keep `generationLoad` (`none`, `low`, or `medium`) and sealed `retrievalTool` as metadata for diagnostic slicing only. Copy `retrievalTool` from the matching sealed blueprint case concept. Read the Execute `retrievalSummary` separately as observed evidence: report its source, confidence, unknown rate, normalized tools, and observed mode when available, but do not replace the sealed design tag, infer missing behavior, or duplicate the observed summary in the scoring record.
6. Validate one judgment per system and case with `validate-judgments.mjs --intake <judge-intake.json>`, then aggregate with `score-results.mjs`. The validator must prove that every judgment's duration exactly matches immutable Execute evidence and that its duration band and adjustment match the contract timeout. Accuracy, already including the duration adjustment, is the primary result. Compare token use only when rounded accuracy is equal and both products have usable token evidence; keep unavailable token values as `null` and never convert them to zero. Compare total speed when token evidence is unavailable or token use differs by at most 5%. Never add another weighted accuracy/token/speed sum.
7. Produce `judgments.json`, `scoreboard.json`, paired `final-report.zh-CN.md` and `final-report.en.md`, and a bilingual audit trail. Keep titles, paragraphs, lists, tables, facts, and decisions corresponding, while writing the Chinese report independently in natural Chinese rather than translating English sentences literally. Include raw scores, pass rates, failures, retries, infrastructure and cleanup issues, token/time evidence, and diagnostic slices across the five capabilities, task type, difficulty, generation load, and sealed `retrievalTool`. Both reports must contain a per-system, per-case table with case ID, final score, actual execution duration in milliseconds plus a human-readable duration, timeout ratio, duration band, duration score adjustment, and completion/comparability status. Retain missing or invalid cases in the table with an unavailable duration or exclusion reason rather than inventing a value. Add a compact observed-retrieval section only when summaries have adequate evidence; show unavailable or low-confidence coverage explicitly. Neither the sealed tag nor the observed mode is a separate score, and no tool choice is mandatory.
8. In a final cleanup gate, verify every result-check `cleanupAudit`, remove Judge-only workspace copies, injected files, processes, temporary credentials, and temporary services, then write `cleanup-manifest.json` from [../../assets/cleanup-manifest.template.json](../../assets/cleanup-manifest.template.json) plus summaries from [../../assets/cleanup-summary.zh-CN.template.md](../../assets/cleanup-summary.zh-CN.template.md) and [../../assets/cleanup-summary.en.template.md](../../assets/cleanup-summary.en.template.md). Set `deletedProductEvidence`, `deletedInitialProjectSnapshots`, and `deletedCompletedStateSnapshots` to false; list every retained initial-project snapshot, completed-state snapshot, product result, and result-check evidence root. Report cleanup never deletes Execute snapshots automatically. Retain product run evidence, judgments, scoreboards, reports, and result-check logs. Report any cleanup failure instead of hiding it.

Generation-load slices are views of the same single score. They are not additional result categories. Explain construct-validity risks when medium-load cases diverge sharply, but do not mathematically subtract or split code-generation ability from the score.
