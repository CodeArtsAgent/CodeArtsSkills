# Execute a Codebase Evalset

Run this workflow inside the product being evaluated, with the evaluated project open. The current agent is the evaluated product's run-level main coordinator. It must not solve cases, judge results, produce a benchmark report, or directly perform case-level prepare/capture/rollback/cleanup. It creates exactly one fresh case-coordinator subagent per case; that coordinator creates one further fresh worker subagent for every attempt.

## E0: mandatory structured intake

Make the first tool call after loading this workflow a structured user-input call. Confirm only information that is not fixed or discoverable:

- evaluated product/system ID and display label;
- round ID as the only shared-layout directory-selection input; fix the evaluation root to `.codebase-eval`, derive every other execution path from the root-owned layout, and run every helper from the evaluated repository root;
- selected public evalset only when more than one approved package is present;
- token-evidence and activity-trace sources only when the product cannot expose them automatically, run-level cost limits only when the user supplied limits, and unresolved environment decisions; allow token evidence to be explicitly `unavailable` instead of inventing a source, and never infer token availability from an activity journal; never ask for a prompt language because evaluated-worker prompts are fixed to English;
- authorization for any exact dependency install or restoration found necessary before execution.

Read the released `execution-contract.json` and verify its SHA-256 and release identity before intake. Apply every shared value from it without asking and without overrides. A requested change to case selection, attempts, concurrency, timeout, English-only worker language, isolation, cleanup, evidence, dependency, service, or credential policy requires a new Generate release; it is not an Execute option.

Derive application-under-evaluation credential requirements and their pre-authoring verification records from the approved package. Do not ask a general credential-availability question and do not request credentials for the evaluated coding product. Re-run least-privilege, redacted readiness checks before creating the TODO; if a required application credential is missing or invalid, stop and request remediation for that concrete requirement. Never persist secret values, and clean sessions or temporary credential state after checks and attempts.

Do not ask for an evaluation root, repository profiling, track rationale, case-count distribution, source authorization, compared products, competitor labels, cross-product token comparability, scoring policy, report audience, an execution run ID, a product-result path, or a report path. Discover released public packages only under `.codebase-eval/<round-id>/evalset/<track>` and ask which track only when multiple approved tracks exist. Never search for a latest evalset or inspect evalset-like artifacts elsewhere, including product-owned configuration/state directories such as `.codeartsdoer`. Generate the execution run ID and derive `.codebase-eval/<round-id>/<system-id>/<execution-run-id>` automatically. After confirmation, create the minimal [execution request](../../assets/execution-request.template.json), pass it to the Runner, and retain the normalized request together with the plan and TODO in one `execution-state.json`. Do not retain separate intake, config, plan, or TODO JSON files.

After the user confirms the round ID, run the resolver from the evaluated repository root with a relative path to the bundled script and `--round <round-id> --system <system-id>`, adding `--track` only after resolving a multi-track choice. Use its returned relative paths verbatim. The resolver fixes the root to `.codebase-eval`; this Execute resolver reads only `round/evalset` and must not enumerate `report` or sibling product directories.

Read [../../references/output-layout.md](../../references/output-layout.md), [../../references/execution-protocol.md](../../references/execution-protocol.md), [../../references/environment-policy.md](../../references/environment-policy.md), [../../references/data-lifecycle.md](../../references/data-lifecycle.md), [../../references/case-schema.md](../../references/case-schema.md), and [../../references/bilingual-output.md](../../references/bilingual-output.md) completely before planning.

## Preconditions

Require this skill package to be installed and discoverable in the evaluated project and an H4-approved public package located under that project. Assume the product supplies the contract-required fresh case-coordinator mechanism, nested fresh-worker mechanism, main-agent-controlled initial snapshot, and case-coordinator-controlled completed-state snapshot and baseline rollback mechanisms; do not ask, probe, or create a preflight capability verdict. Never execute cases in the main agent or case coordinator.

Run environment discovery before creating the TODO. If installation or restoration is required, ask for the exact action, complete it only after approval, verify it, and seal the environment. For every selected-case service, repeat the planned build/start/application-readiness/stop/cleanup lifecycle and create a candidate-safe `environmentPolicy.serviceReadiness` record only after it passes. Start no case until every selected case is dependency-ready and every named service has a verified record. No participant may mutate dependencies after the first case starts.

Enforce both subagent layers and disclosure behavior at runtime. Immediately stop the entire run, terminate the active worker and case coordinator, restore the project baseline, clean attempt state, and record a protocol error when either subagent inherits parent context, lacks a unique agent/session identity, the worker receives more than one case, reads the evaluation root or another case, is not spawned by the assigned case coordinator, writes outside the evaluated project or declared external scopes, or when the main coordinator or case coordinator solves or repairs a case. Do not continue with another case after such a violation.

## Evidence collection contract

Collect sufficient, dimension-neutral evidence so the later Judge can assess every reporting dimension without seeing the evaluated session live:

- final answer and all candidate-created deliverables;
- baseline/final inventories, changed paths, file hashes, and the preserved frozen workspace;
- a bounded chronological key-activity journal covering material searches, reads, edits, commands, and tests; collapse repetitive operations and do not copy a full product-native tool transcript;
- command and test outputs with exit status or failure evidence;
- repository paths and symbols explicitly cited or inspected when the host exposes them;
- child/session identity, timestamps, token evidence, timeout/unfinished state, environment baseline, retries, and protocol violations.

Prefer a host-native transcript/tool trace as a transient source, then reduce it to the bounded key-activity journal and at most eight representative retrieval actions; do not retain the full trace solely for retrieval analysis. When native evidence is unavailable, require the worker to maintain the standardized `activity-log.jsonl` and mark retrieval evidence as `worker-journal`. Each representative action contains only normalized tool, `discovery` or `verification` role, target, and observable outcome. The deterministic Runner derives unique tools, lexical/index/unknown discovery counts, observed mode, and confidence. A file read used only to inspect an index result is `verification` and does not independently turn `codebase-index` into `hybrid`. Never request or record hidden chain-of-thought.

This evidence supports later accuracy, retrieval, architecture, workflow, impact, diagnosis, verification, token, and speed analysis. Do not expose sealed case annotations or scoring criteria and do not translate evidence into scores.

## Agent protocol

1. Validate the shared layout and round identity, reserve `.codebase-eval/<round-id>/<system-id>/<execution-run-id>`, and initialize one run for this evaluated product. Maintain the sole authoritative `execution-state.json`; its `todo` section is the only mutable run state. Put each verified, credential-free startup contract needed by selected cases in `todo.serviceReadiness`; the main coordinator and current case coordinator may read and operate from this shared section. Never inspect another round, `report/`, or sibling product directories.
2. Read only the public manifest and case index for scheduling. Do not pre-read or summarize case prompts.
3. Before the first attempt, the main coordinator invokes `project-state.mjs snapshot` to create `project-baseline-snapshot/` and `project-baseline-manifest.json` under its run root. The ordinary snapshot inventory includes tracked project files and untracked files not ignored by Git, with modes and symbolic links; it excludes `.codebase-eval`, the case protocol directory, `.git`, and every file or directory ignored through Git's standard ignore rules. Preserve `.git` separately inside the private baseline snapshot for restoration, but never compare its files byte for byte. Record and verify semantic Git state using HEAD, symbolic HEAD, refs, and staged index entries. Never assume a clean Git worktree.
4. Select one pending case by ID and spawn exactly one fresh case-coordinator subagent with no inherited conversation (`fork_turns: "none"` or equivalent). Give it only the run-root-relative control path, task ID, and English lifecycle instructions; never give it case prompt content, solving hints, another case, or Judge material. Immediately record its unique agent/session IDs through Runner `coordinate`. Only one case coordinator may be active, and it remains the sole coordinator for that case across an allowed infrastructure retry.
5. The case coordinator invokes Runner `prepare` to prove the live project still matches the initial baseline and materialize exactly one English `worker-envelope.json`. Include only English title, prompt, deliverables, allowed/forbidden operations, environment notes, worker instructions, and case-scoped service notes. Keep `.codebase-eval`, the complete TODO, every other case, prior results, reports, sibling product material, private material, Chinese comparison text, and future Git history unavailable to the worker through product access controls or reversible coordinator visibility controls.
6. The case coordinator spawns one further fresh worker with no inherited conversation. Give it only the envelope path and instruct it to execute that one task directly in the evaluated project, following the public task's allowed and forbidden operations exactly. For schema-v4 read-only cases, the worker must not modify source, run project tests or builds, or start an application or service. Immediately record its unique worker agent/session IDs through Runner `assign`. The worker itself records `startedAt` immediately before beginning the case and `completedAt` immediately after finishing, using second-precision UTC timestamps in the exact form `YYYY-MM-DDTHH:mm:ssZ`. The coordinator must not synthesize, round, or rewrite these timestamps.
7. Run exactly one case coordinator and at most one nested worker at a time. Do not coordinate, prepare, assign, run, capture, retry, or clean another case while the current case is nonterminal. Enforce the one-hour timeout on each worker attempt; lifecycle gates must complete before any retry or next case.
8. On worker return, the case coordinator first invokes `project-state.mjs freeze --project . --snapshot <completed-state-workspace>`. It then captures the diff, final response, bounded activity/command/test and retrieval evidence, cited or inspected locations, the worker-recorded start/end timestamps, worker token/duration evidence, both subagent identities, evidence gaps, and violations. Treat the worker answer, project output, timestamps, activity journal, and observable evidence as immutable. Require the worker to create or replace `final-response.md` with only the active case's answer; never append or replay host/chat output, and never concatenate an earlier case response. Every repository location used as evidence in the answer must use its complete repository-root-relative path, optionally with a line range or symbol, and must also appear in `citedRepositoryLocations`; locations that were inspected but not disclosed remain only in `inspectedRepositoryLocations`. Runner capture validates citation resolution and exact appearance in the final response, all other sources, the live-versus-frozen inventory, and the absence of a complete prior response appended as a suffix before atomically publishing `final-response.md` and `execution-result.json`. A detected cross-case suffix terminates the run as a non-retryable protocol violation and is not converted into a scored answer. The execution result must retain `startedAt` and `completedAt` exactly as supplied by the worker.
9. After completed-state verification, the case coordinator invokes `project-state.mjs restore` against the immutable initial snapshot and supplies transient `rollback-result.json`. The Runner verifies the restored worktree byte for byte, Git state semantically, `.codebase-eval-worker` removal, and the preserved completed workspace. Do not start cleanup until rollback is verified.
10. The case coordinator terminates the attempt process tree, runs only pre-approved external cleanup controls, removes declared external/cache/temp state, verifies the sealed environment baseline, and supplies transient `cleanup-result.json`. Continue only when cleanup is `clean` with no dirty data remaining.
11. On `rollback-error` or `cleanup-error`, stop the entire run and request human remediation. After remediation, discard the contaminated run and restart every case from a newly sealed baseline.
12. Retry only an infrastructure failure after rollback and cleanup pass. Keep the same case coordinator, but create a new fresh worker. Coordinator lifecycle tokens and time are protocol overhead; candidate token/time metrics come only from worker attempts. Coordinator, worker, or disclosure violations terminate the run and are not retryable. Never retry an incorrect or low-quality answer.
13. If a missing dependency is discovered mid-run, stop the whole run. After approved remediation, discard the run and restart every case from a newly sealed environment.
14. When all TODO items are terminal and every case coordinator has returned, the main coordinator aggregates `execution-results.json` and validates the run. Do not run private tests, assign scores, interpret quality, or create a report.

Both subagent layers are supplied by the evaluated product, but only the worker's code changes, answer, worker timestamps, worker tokens, and worker duration are measured candidate output. The case coordinator performs protocol mechanics and may validate artifact shape, but may not repair, complete, reinterpret, or improve the worker's work.

## Completion output

Store only the durable evaluated output under `.codebase-eval/<round-id>/<system-id>/<execution-run-id>`: consolidated `execution-state.json`, one independent `execution-result.json` per attempted case, `final-response.md`, bounded key-activity and retrieval summaries, command/test/deliverable evidence, preserved completed-state workspaces, and `execution-results.json` with embedded validation. JSON is reserved for these minimal machine interfaces; human answers remain Markdown and observable activity remains JSON Lines. Do not retain capture-result, restoration-result, cleanup-result, a full product-native trace collected solely for retrieval analysis, temporary inventory, or gate-only log files after their checks pass or fail. Do not create separate intake/config/plan/TODO/validation JSON files. A short bilingual completion receipt may list artifact paths, evidence gaps, compact integrity state, and incomplete infrastructure states; it must contain no score, case-quality conclusion, or comparison.
